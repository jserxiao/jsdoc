# 微前端与大型应用架构

## 概述

解决大型应用拆分、多团队协作、遗留系统迁移等场景的需求。随着业务复杂度提升，微前端已成为大型前端项目的标准架构方案。

## 微前端框架

### qiankun
- **定位**：乾坤微前端框架，基于 single-spa
- **特点**：完善沙箱、样式隔离、开箱即用
- **适用场景**：React/Vue 主应用、需要完整解决方案

### single-spa
- **定位**：微前端基石框架
- **特点**：框架无关、高度灵活、生态丰富
- **适用场景**：需要高度定制的微前端架构

### MicroApp
- **定位**：基于 WebComponent 的微前端
- **特点**：使用简单、沙箱隔离、接入成本低
- **适用场景**：快速接入微前端、降低改造成本

### wujie
- **定位**：无界微前端，WebComponent + iframe
- **特点**：完美沙箱、支持子应用保活、接入简单
- **适用场景**：需要完善隔离、子应用独立运行

## 模块联邦

### Module Federation
- **定位**：Webpack 5 模块联邦
- **特点**：运行时共享模块、独立部署、版本管理
- **适用场景**：共享组件库、独立团队开发

### @module-federation/enhanced
- **定位**：增强版模块联邦
- **特点**：更好的 TypeScript 支持、更多构建工具适配
- **适用场景**：现代化项目、跨构建工具共享

## Monorepo 管理

### pnpm workspace
- **定位**：轻量级 Monorepo
- **特点**：依赖提升、磁盘空间节省、性能优秀
- **适用场景**：中小型 Monorepo、快速上手

### Turborepo
- **定位**：高性能构建系统
- **特点**：智能缓存、并行构建、任务编排
- **适用场景**：大型 Monorepo、CI/CD 优化

### Nx
- **定位**：智能 Monorepo 工具
- **特点**：依赖图分析、增量构建、代码生成
- **适用场景**：企业级 Monorepo、需要完整工具链

### Lerna
- **定位**：Monorepo 版本管理
- **特点**：版本管理、发布流程、变更日志
- **适用场景**：多包发布、独立版本管理

## 大型应用状态管理

### Redux Toolkit
- **定位**：大型应用状态管理首选
- **特点**：标准化最佳实践、强大的中间件、DevTools
- **适用场景**：复杂状态逻辑、需要可预测性

### Pinia
- **定位**：Vue 大型应用状态管理
- **特点**：模块化、TypeScript 友好、组合式 API
- **适用场景**：Vue 3 大型应用

### XState
- **定位**：状态机与状态图
- **特点**：可视化、可测试、复杂状态建模
- **适用场景**：复杂业务流程、多状态转换

## 架构设计原则

### 拆分策略
- **按业务域拆分**：不同业务独立子应用
- **按团队拆分**：团队边界即应用边界
- **按技术栈拆分**：兼容不同技术栈

### 通信方案
- **props/emit**：主子应用通信
- **自定义事件**：跨应用通信
- **共享状态**：全局状态管理
- **URL 参数**：路由级别通信

### 样式隔离
- **CSS Modules**：编译时隔离
- **CSS-in-JS**：运行时隔离
- **Shadow DOM**：浏览器原生隔离
- **命名空间**：约定前缀隔离

## 技术选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 遗留系统改造 | qiankun | 接入成本低、完善沙箱 |
| 新建大型应用 | wujie/MicroApp | 接入简单、隔离完善 |
| 多团队协作 | Monorepo + Module Federation | 独立开发部署 |
| 组件库共享 | Module Federation | 运行时共享、版本独立 |
| 库开发 | pnpm workspace | 轻量高效 |

## 常见问题

### 样式冲突
- 使用 CSS Modules 或 CSS-in-JS
- 配置样式隔离（qiankun sandbox）
- 使用命名空间前缀

### 状态共享
- 避免全局污染
- 使用事件通信
- 设计清晰的接口契约

### 性能优化
- 预加载子应用
- 合理拆分粒度
- 缓存策略优化
