# HTTP 基础概念

> HTTP（HyperText Transfer Protocol，超文本传输协议）是 Web 的核心协议，定义了客户端与服务器之间的通信规则。深入理解 HTTP 协议对于前端开发、性能优化和问题排查至关重要。

## 学习要点

- 🌐 理解 HTTP 协议的工作原理和请求响应模型
- 📦 掌握 HTTP 协议栈和各层关系
- 🔄 了解 HTTP 版本的演进历程
- 🔐 理解 HTTP 与 HTTPS 的区别和安全机制

---

## 1. 什么是 HTTP

HTTP 是一种**无状态的、基于请求-响应模式的应用层协议**，运行在 TCP/IP 协议栈之上。

### 1.1 协议栈位置

```
应用层    HTTP、HTTPS、WebSocket
           ↓
传输层    TCP、UDP
           ↓
网络层    IP
           ↓
链路层    Ethernet、WiFi
```

### 1.2 HTTP 的特点

```
┌─────────────────────────────────────────────────────────────────────────┐
│  HTTP 协议特点                                                            │
├────────────────┬────────────────────────────────────────────────────────┤
│  无状态         │  服务器不保存客户端状态，每次请求独立                    │
│  无连接         │  每次请求-响应后断开连接（HTTP/1.1 后支持持久连接）       │
│  基于请求-响应   │  客户端发起请求，服务器返回响应                          │
│  简单快速       │  请求方法简单，通信速度快                               │
│  灵活           │  可传输任意类型数据（Content-Type 标识）                  │
└────────────────┴────────────────────────────────────────────────────────┘
```

### 1.3 无状态的影响

```
优点：
┌─────────────────────────────────────────────────────────────────────────┐
│  • 服务器实现简单，不需要维护连接状态                                      │
│  • 服务器可以轻松扩展，支持更多客户端                                      │
│  • 请求失败后可以重试，不影响其他请求                                      │
└─────────────────────────────────────────────────────────────────────────┘

缺点：
┌─────────────────────────────────────────────────────────────────────────┐
│  • 每次请求都需要携带完整的认证信息                                        │
│  • 无法记住用户状态（需要 Cookie/Session/Token 机制）                      │
│  • 无法实现推送（需要 WebSocket/Server-Sent Events）                      │
└─────────────────────────────────────────────────────────────────────────┘
```

## 2. HTTP 版本演进详解

### 2.1 版本发展历程

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│                           HTTP 协议发展历程                                             │
├──────────┬──────────────┬────────────────────────────────────────────────────────────────┤
│  版本     │  年份         │  里程碑意义                                                       │
├──────────┼──────────────┼────────────────────────────────────────────────────────────────┤
│  HTTP/0.9│  1991        │  蒂姆·伯纳斯-李发明，用于 WWW 项目，最简单的文本传输协议            │
│  HTTP/1.0│  1996.05     │  RFC 1945，首次标准化，引入头部和多种方法，奠定现代 HTTP 基础        │
│  HTTP/1.1│  1997.01     │  RFC 2068（1997）→ RFC 2616（1999）→ RFC 7230-7235（2014）        │
│          │              │  持久连接、管道化、Host 头部、分块传输，成为主流标准 20+ 年         │
│  HTTP/2  │  2015.05     │  RFC 7540，Google SPDY 协议演进而来，二进制分帧、多路复用            │
│  HTTP/3  │  2022.06     │  RFC 9114，基于 QUIC 协议（UDP），彻底解决队头阻塞，更快连接建立     │
└──────────┴──────────────┴────────────────────────────────────────────────────────────────┘
```

### 2.2 HTTP/0.9 - 原始版本（1991）

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HTTP/0.9 - 单行协议                                                                  │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  设计理念                                                                              │
│  • 极简设计：只满足 HTML 文档传输需求                                                   │
│  • 无状态：每次请求独立，服务器不保存任何信息                                            │
│  • 单次连接：请求-响应后立即断开 TCP 连接                                               │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  核心特点                                                                              │
│  • 只支持 GET 方法                                                                     │
│  • 没有请求头和响应头                                                                   │
│  • 没有 Status Line（状态行）                                                          │
│  • 只能传输 HTML 文本（不支持其他 MIME 类型）                                            │
│  • 没有状态码（成功返回内容，失败返回包含错误信息的 HTML）                                 │
│  • 使用 ASCII 文本传输                                                                 │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  请求格式                                                                              │
│  GET <path>                                                                            │
│  示例：GET /index.html                                                                 │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  响应格式                                                                              │
│  <html>...</html>（直接返回 HTML 内容，连接关闭）                                        │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  完整通信示例                                                                          │
│  客户端 → 服务器：                                                                     │
│  GET /index.html                                                                       │
│                                                                                        │
│  服务器 → 客户端：                                                                     │
│  <html>                                                                                │
│    <head><title>Hello</title></head>                                                   │
│    <body>Hello World</body>                                                            │
│  </html>                                                                               │
│  [连接关闭]                                                                            │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  局限性                                                                                │
│  • 无法传输图片、视频、CSS、JavaScript 等资源                                           │
│  • 无法传递元信息（字符编码、内容类型等）                                                │
│  • 无法进行缓存协商                                                                     │
│  • 每次请求都建立新连接，性能极差                                                        │
│  • 无法区分成功和失败                                                                   │
└───────────────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 HTTP/1.0 - 标准化起步（1996）

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HTTP/1.0 - 构建可扩展性                                                               │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  RFC 文档：RFC 1945（Informational）                                                   │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  重大改进                                                                              │
│  ────────────────────────────────────────────────────────────────────────────────────│
│  1. 协议版本号                                                                         │
│     • 请求行包含 HTTP 版本：GET /index.html HTTP/1.0                                   │
│                                                                                        │
│  2. 请求头和响应头                                                                     │
│     • 引入 Header 机制，可传递元信息                                                    │
│     • 格式：Name: Value<CRLF>，以空行结束                                               │
│                                                                                        │
│  3. 新增方法                                                                           │
│     • GET：获取资源                                                                   │
│     • POST：提交数据（表单、文件上传）                                                  │
│     • HEAD：只获取响应头，不获取响应体                                                  │
│                                                                                        │
│  4. 状态码和原因短语                                                                   │
│     • Status Line 格式：HTTP/1.0 <status-code> <reason-phrase>                         │
│     • 示例：HTTP/1.0 200 OK, HTTP/1.0 404 Not Found                                    │
│                                                                                        │
│  5. Content-Type 支持                                                                  │
│     • 支持多种 MIME 类型：text/html, image/jpeg, application/pdf 等                   │
│     • 可传输图片、视频、二进制文件                                                      │
│                                                                                        │
│  6. Content-Length                                                                    │
│     • 声明响应体长度，支持二进制传输                                                    │
│                                                                                        │
│  7. 其他头部                                                                           │
│     • User-Agent：客户端标识                                                          │
│     • Accept：内容协商                                                                 │
│     • Authorization：认证信息                                                          │
│     • Referer：来源页面                                                                │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  完整请求示例                                                                          │
│  GET /index.html HTTP/1.0                                                              │
│  User-Agent: Mozilla/5.0                                                              │
│  Accept: text/html, image/gif, image/jpeg, */*                                         │
│                                                                                        │
│  响应示例                                                                              │
│  HTTP/1.0 200 OK                                                                       │
│  Date: Tue, 15 Nov 1996 08:12:31 GMT                                                  │
│  Server: CERN/3.0 libwww/2.17                                                         │
│  Content-Type: text/html                                                              │
│  Content-Length: 3456                                                                 │
│                                                                                        │
│  <html>...</html>                                                                      │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  连接模型                                                                              │
│  • 默认：每次请求建立新的 TCP 连接，响应后关闭                                           │
│  • 可选持久连接：Connection: keep-alive（非标准扩展）                                   │
│                                                                                        │
│  TCP 连接开销：                                                                        │
│  客户端                    服务器                                                       │
│    │                         │                                                         │
│    │──── SYN ────────────────►│  （1 RTT）                                             │
│    │◄─── SYN+ACK ────────────│                                                         │
│    │──── ACK ────────────────►│                                                        │
│    │                         │                                                         │
│    │──── HTTP Request ──────►│                                                         │
│    │◄─── HTTP Response ─────│                                                         │
│    │                         │                                                         │
│    │──── FIN ────────────────►│  （关闭连接）                                          │
│    │◄─── FIN+ACK ────────────│                                                         │
│    │──── ACK ────────────────►│                                                        │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  主要缺陷                                                                              │
│  • 每次请求都需要 TCP 三次握手 + 慢启动，性能差                                          │
│  • 并发请求需要建立多个 TCP 连接                                                        │
│  • Host 头部缺失，不支持虚拟主机                                                        │
│  • 缓存机制不完善（只有 Expires 和 Pragma）                                             │
│  • 缺少持久连接的标准支持                                                               │
└───────────────────────────────────────────────────────────────────────────────────────┘
```

### 2.4 HTTP/1.1 - 主流标准（1997-2014）

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HTTP/1.1 - 性能与功能完善                                                             │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  RFC 文档                                                                              │
│  • RFC 2068（1997.01）：HTTP/1.1 初版                                                  │
│  • RFC 2616（1999.06）：HTTP/1.1 修订版（广泛采用）                                     │
│  • RFC 7230-7235（2014.06）：HTTP/1.1 语义拆分修订                                      │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  核心改进                                                                              │
│  ────────────────────────────────────────────────────────────────────────────────────│
│  1. 持久连接（Persistent Connection）                                                  │
│     • 默认启用：Connection: keep-alive                                                 │
│     • 一次 TCP 连接可发送多个请求                                                       │
│     • 大幅减少连接建立开销                                                              │
│     • Keep-Alive 头部控制参数：timeout, max                                            │
│                                                                                        │
│  2. 管道化（Pipelining）                                                               │
│     • 可连续发送多个请求，无需等待响应                                                   │
│     • 响应必须按请求顺序返回（FIFO）                                                    │
│     • 问题：存在队头阻塞（Head-of-Line Blocking）                                       │
│     • 现状：浏览器默认禁用，实际很少使用                                                 │
│                                                                                        │
│  3. Host 头部（必需）                                                                  │
│     • 一个 IP 地址可托管多个域名（虚拟主机）                                            │
│     • 支持共享主机和 CDN 部署                                                          │
│     • HTTP/1.1 必须包含 Host 头部                                                       │
│                                                                                        │
│  4. 分块传输编码（Chunked Transfer Encoding）                                          │
│     • Transfer-Encoding: chunked                                                       │
│     • 无需预先知道 Content-Length                                                      │
│     • 支持动态生成内容流式传输                                                          │
│                                                                                        │
│  5. 新增方法                                                                           │
│     • PUT：更新资源                                                                   │
│     • DELETE：删除资源                                                                │
│     • OPTIONS：查询支持的方法                                                          │
│     • TRACE：诊断请求链路                                                              │
│     • CONNECT：建立隧道（HTTPS 代理）                                                  │
│                                                                                        │
│  6. 缓存增强                                                                           │
│     • ETag / If-None-Match：实体标签验证                                              │
│     • If-Modified-Since / Last-Modified：时间验证                                     │
│     • Cache-Control：更灵活的缓存控制指令                                              │
│     • Vary：区分缓存版本                                                               │
│                                                                                        │
│  7. 范围请求（Range Request）                                                          │
│     • Range: bytes=0-1023                                                             │
│     • Accept-Ranges: bytes                                                            │
│     • 支持断点续传、视频拖动                                                            │
│                                                                                        │
│  8. 内容协商                                                                           │
│     • Accept, Accept-Language, Accept-Encoding, Accept-Charset                        │
│     • 服务器根据客户端偏好返回最适合的内容                                               │
│                                                                                        │
│  9. 压缩传输                                                                           │
│     • Content-Encoding: gzip / deflate                                                │
│     • 减少传输数据量                                                                   │
│                                                                                        │
│  10. 身份认证增强                                                                      │
│      • Digest 认证                                                                    │
│      • 改进的 Authorization 头部                                                       │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  持久连接工作流程                                                                      │
│  客户端                    服务器                                                       │
│    │                         │                                                         │
│    │──── TCP 三次握手 ──────►│                                                         │
│    │                         │                                                         │
│    │──── Request 1 ─────────►│                                                         │
│    │◄─── Response 1 ────────│  ◄── 连接保持 ──►                                        │
│    │──── Request 2 ─────────►│                                                         │
│    │◄─── Response 2 ────────│                                                         │
│    │──── Request 3 ─────────►│                                                         │
│    │◄─── Response 3 ────────│                                                         │
│    │         ...            │                                                         │
│    │──── Connection: close ─►│                                                         │
│    │◄─── TCP 关闭 ──────────│                                                         │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  队头阻塞问题图解                                                                      │
│                                                                                        │
│  管道化请求（请求可并行发送，但响应必须顺序返回）：                                      │
│  请求1 ────────────────────────────────────────►                                       │
│  请求2 ────►                                                                           │
│  请求3 ────►                                                                           │
│           等待...                                                                       │
│                          响应1 ◄────────────────────────────────────────               │
│                          响应2 ◄─────────── （请求2和3被请求1阻塞）                     │
│                          响应3 ◄───────────                                            │
│                                                                                        │
│  如果请求1 处理时间长，后续请求必须等待！                                               │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  HTTP/1.1 并发解决方案                                                                 │
│  • 域名分片（Domain Sharding）：使用多个子域名绕过浏览器连接限制                          │
│    - 浏览器对单个域名并发连接数限制（通常 6 个）                                         │
│    - static1.example.com, static2.example.com, ...                                    │
│  • 资源合并：CSS/JS 文件合并，减少请求数                                                │
│  • 雪碧图：多个小图标合并为一张大图                                                     │
│  • 内联资源：小图片转 Base64 内联到 CSS/HTML                                            │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  完整请求示例                                                                          │
│  GET /api/users HTTP/1.1                                                               │
│  Host: www.example.com                                                                │
│  User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)                                │
│  Accept: application/json, text/plain, */*                                             │
│  Accept-Language: zh-CN,zh;q=0.9,en;q=0.8                                             │
│  Accept-Encoding: gzip, deflate                                                        │
│  Connection: keep-alive                                                               │
│  Cache-Control: max-age=0                                                             │
│  If-None-Match: "33a64df551425fcc55e"                                                 │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  局限性                                                                                │
│  • 队头阻塞：管道化未能解决问题，实际被禁用                                              │
│  • 头部冗余：每次请求都携带完整头部，大量重复                                            │
│  • 连接数限制：浏览器对单个域名并发连接数有限制                                          │
│  • 文本协议：解析效率低，安全性差                                                       │
│  • TCP 队头阻塞：丢包导致后续数据等待重传                                               │
└───────────────────────────────────────────────────────────────────────────────────────┘
```

### 2.5 HTTP/2 - 性能飞跃（2015）

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HTTP/2 - 多路复用与二进制分帧                                                         │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  RFC 文档：RFC 7540（2015.05）                                                         │
│  背景：Google SPDY 协议演进而来（SPDY 于 2016 年废弃）                                   │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  核心特性                                                                              │
│  ────────────────────────────────────────────────────────────────────────────────────│
│  1. 二进制分帧（Binary Framing）                                                       │
│     ┌──────────────────────────────────────────────────────────────────────────────┐│
│     │  帧（Frame）是 HTTP/2 通信的最小单位                                           ││
│     │  ┌─────────────────────────────────────────────────────────────────────────┐ ││
│     │  │  Length (24)  │  Type (8)  │  Flags (8)  │  R (1)  │  Stream ID (31)   │ ││
│     │  └─────────────────────────────────────────────────────────────────────────┘ ││
│     │  │                           Frame Payload                                   │ ││
│     │  └─────────────────────────────────────────────────────────────────────────┘ ││
│     └──────────────────────────────────────────────────────────────────────────────┘│
│     • 帧类型：DATA, HEADERS, PRIORITY, RST_STREAM, SETTINGS, PUSH_PROMISE, etc.      │
│     • 更高效解析，更少解析错误                                                          │
│     • 更好的网络传输效率                                                                │
│                                                                                        │
│  2. 多路复用（Multiplexing）                                                           │
│     • 单个 TCP 连接并行发送多个请求/响应                                                │
│     • 每个请求/响应是一个流（Stream）                                                   │
│     • 流被分解为多个帧，交错传输                                                        │
│     • 完全解决应用层队头阻塞                                                            │
│     • 每个流有唯一 Stream ID，客户端发起为奇数，服务器发起为偶数                         │
│                                                                                        │
│  3. 头部压缩（HPACK）                                                                  │
│     ┌──────────────────────────────────────────────────────────────────────────────┐│
│     │  压缩机制：                                                                    ││
│     │  • 静态字典（Static Table）：预定义的常用头部键值对                            ││
│     │    - :method: GET → 索引 2                                                   ││
│     │    - :path: / → 索引 4                                                       ││
│     │    - :scheme: https → 索引 7                                                 ││
│     │  • 动态字典（Dynamic Table）：连接期间学习的头部键值对                          ││
│     │  • Huffman 编码：对头部值进行压缩                                             ││
│     └──────────────────────────────────────────────────────────────────────────────┘│
│     • 减少 80-90% 头部数据量                                                           │
│     • 大幅提升传输效率                                                                 │
│                                                                                        │
│  4. 服务器推送（Server Push）                                                          │
│     • 服务器主动推送客户端可能需要的资源                                                │
│     • PUSH_PROMISE 帧通知客户端即将推送                                                │
│     • 减少请求延迟，提前加载关键资源                                                    │
│     • 示例：请求 HTML 时主动推送 CSS 和 JS                                             │
│     • 注意：Chrome 已移除服务器推送支持（2022），推荐使用 103 Early Hints               │
│                                                                                        │
│  5. 请求优先级                                                                         │
│     • 每个流可设置权重（1-256）和依赖关系                                               │

### 2.6 HTTP/3 - 未来标准（2022）

```
┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HTTP/3 - QUIC 协议革新                                                               │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  RFC 文档：RFC 9114（2022.06）                                                         │
│  背景：Google QUIC 协议演进而来                                                        │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  核心变革                                                                              │
│  ────────────────────────────────────────────────────────────────────────────────────│
│  1. 基于 QUIC 协议（UDP）                                                              │
│     ┌──────────────────────────────────────────────────────────────────────────────┐│
│     │  协议栈对比：                                                                  ││
│     │  HTTP/1.1, HTTP/2：HTTP → TLS → TCP → IP                                     ││
│     │  HTTP/3：           HTTP/3 → QUIC → UDP → IP                                 ││
│     └──────────────────────────────────────────────────────────────────────────────┘│
│     • QUIC = Quick UDP Internet Connections                                           │
│     • 在用户态实现可靠传输，避免内核态 TCP 的限制                                       │
│     • 彻底解决 TCP 层面的队头阻塞                                                      │
│     • 更好的移动端切换体验                                                             │
│                                                                                        │
│  2. 0-RTT 连接建立                                                                    │
│     ┌──────────────────────────────────────────────────────────────────────────────┐│
│     │  连接建立时间对比：                                                            ││
│     │  HTTP/1.1 over TLS 1.3：TCP 1-RTT + TLS 1-RTT = 2 RTT                        ││
│     │  HTTP/2 over TLS 1.3：  TCP 1-RTT + TLS 1-RTT = 2 RTT                        ││
│     │  HTTP/3 (首次)：        QUIC 1-RTT（包含加密握手）                             ││
│     │  HTTP/3 (复用)：        0-RTT（立即发送请求）                                  ││
│     └──────────────────────────────────────────────────────────────────────────────┘│
│     • 复用之前的连接信息（Session Ticket）                                             │
│     • 立即发送应用数据                                                                 │
│     • 极快的连接恢复和冷启动                                                           │
│                                                                                        │
│  3. 独立流传输（解决 TCP 队头阻塞）                                                    │
│     ┌──────────────────────────────────────────────────────────────────────────────┐│
│     │  TCP 队头阻塞（HTTP/2）：                                                     ││
│     │  流1: [包1] [包2] [包3-丢失] [包4-等待] [包5-等待]                            ││
│     │  流2:                    [阻塞等待包3重传]                                    ││
│     │  流3:                    [阻塞等待包3重传]                                    ││
│     │                                                                                ││
│     │  QUIC 独立流（HTTP/3）：                                                       ││
│     │  流1: [包1] [包2] [包3-丢失] [包4-等待] [包5-等待]                            ││
│     │  流2:               [正常传输] [正常传输] [正常传输]                           ││
│     │  流3:               [正常传输] [正常传输] [正常传输]                           ││
│     └──────────────────────────────────────────────────────────────────────────────┘│
│     • 每个流独立传输，丢包只影响单个流                                                  │
│     • 大幅提升弱网环境下的性能                                                         │
│                                                                                        │
│  4. 连接迁移（Connection Migration）                                                  │
│     • 使用 Connection ID 标识连接，而非四元组（IP+端口）                                │
│     • 网络切换（WiFi → 4G）不断开连接                                                  │
│     • 移动端体验大幅提升                                                               │
│     • 无需重新建立连接和握手                                                           │
│                                                                                        │
│  5. 改进的拥塞控制                                                                     │
│     • 可插拔的拥塞控制算法                                                             │
│     • 更快的收敛和恢复                                                                │
│     • 更适合现代网络环境                                                               │
│                                                                                        │
│  6. 前向纠错（Forward Error Correction, FEC）                                         │
│     • 发送冗余数据包                                                                   │
│     • 接收方可直接恢复丢失数据，无需重传                                               │
│     • 进一步降低延迟                                                                   │
│                                                                                        │
│  7. QPACK 头部压缩                                                                    │
│     • HTTP/2 HPACK 的改进版本                                                         │
│     • 解决 HPACK 在乱序传输时的队头阻塞问题                                            │
│     • 使用静态表和动态表，支持乱序编码/解码                                            │
│                                                                                        │
│  8. 内置安全                                                                           │
│     • QUIC 强制使用 TLS 1.3                                                           │
│     • 加密所有握手和元数据                                                            │
│     • 更好的隐私保护                                                                   │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  QUIC 帧类型                                                                          │
│  • STREAM：数据流帧                                                                   │
│  • ACK：确认帧                                                                        │
│  • CRYPTO：加密握手帧                                                                 │
│  • CONNECTION_CLOSE：连接关闭帧                                                       │
│  • DATAGRAM：数据报帧                                                                 │
│  • NEW_CONNECTION_ID：新连接ID帧                                                      │
│  • PATH_CHALLENGE / PATH_RESPONSE：路径验证帧                                          │
│  • RETIRE_CONNECTION_ID：退役连接ID帧                                                  │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  HTTP/2 vs HTTP/3 性能对比                                                            │
│  ┌───────────────────────────────────────────────────────────────────────────────────┐│
│  │  指标            │  HTTP/2                   │  HTTP/3                           ││
│  ├──────────────────┼─────────────────────────┼────────────────────────────────────┤│
│  │  传输层          │  TCP                     │  UDP (QUIC)                        ││
│  │  连接建立        │  2 RTT (TCP+TLS)         │  1 RTT / 0 RTT (复用)              ││
│  │  队头阻塞        │  TCP 层存在              │  完全解决                          ││
│  │  连接迁移        │  不支持                  │  支持                              ││
│  │  头部压缩        │  HPACK                   │  QPACK                             ││
│  │  丢包恢复        │  TCP 重传（阻塞）         │  独立流恢复                        ││
│  │  移动端体验      │  网络切换断开            │  无缝切换                          ││
│  │  隐私保护        │  部分元数据明文           │  全加密                            ││
│  └──────────────────┴─────────────────────────┴────────────────────────────────────┘│
├───────────────────────────────────────────────────────────────────────────────────────┤
│  连接迁移示例                                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────────────┐ │
│  │  场景：用户从 WiFi 切换到 4G                                                     │ │
│  │                                                                                  │ │
│  │  TCP (HTTP/2)：                                                                 │ │

│  └─────────────────────────────────────────────────────────────────────────────────┘ │
├───────────────────────────────────────────────────────────────────────────────────────┤

│  • Firefox：支持（默认启用）                                                          │

│  • Apache：2.4.57+ 支持                                                               │
│  • Caddy：原生支持                                                                    │
│  • Cloudflare：全面支持                                                               │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  局限性                                                                                │
│  • UDP 可能被网络设备限制或 QoS 降级                                                   │

│  • 监控性能：使用 Navigation Timing API 和 Resource Timing API 评估效果                 │
└───────────────────────────────────────────────────────────────────────────────────────┘
```

## 3. HTTP 与 HTTPS

### 3.1 区别对比

```
HTTP（明文传输）：
┌──────┐     明文      ┌──────┐     明文      ┌──────────┐
│ 客户端│ ──────────► │ 代理  │ ──────────► │   服务器  │
└──────┘              └──────┘              └──────────┘
           ⚠️ 可被窃听、篡改、冒充

HTTPS（加密传输）：
┌──────┐    加密     ┌──────┐    加密     ┌──────────┐
│ 客户端│ ─────────► │ 代理  │ ─────────► │   服务器  │
└──────┘            └──────┘            └──────────┘
           ✓ TLS/SSL 加密
           ✓ 身份认证（证书）
           ✓ 数据完整性
```

```
┌─────────────────┬─────────────────────────┬─────────────────────────┐
│  对比项          │  HTTP                    │  HTTPS                   │
├─────────────────┼─────────────────────────┼─────────────────────────┤
│  端口           │  80                      │  443                     │
│  传输方式       │  明文                    │  加密（TLS/SSL）          │
│  安全性         │  低（可被窃听、篡改）       │  高（加密、认证、完整性）   │
│  证书           │  不需要                  │  需要数字证书              │
│  性能           │  快                      │  略慢（握手开销）          │
│  SEO            │  无优势                  │  搜索引擎优先收录          │
│  浏览器标识      │  不显示安全标志           │  显示锁形标志              │
└─────────────────┴─────────────────────────┴─────────────────────────┘
```

### 3.2 HTTPS 工作原理

#### TLS/SSL 握手过程

```
客户端                                    服务器
  │                                         │
  │  1. Client Hello                        │
  │  (支持的加密套件、随机数)                  │
  │ ─────────────────────────────────────►  │
  │                                         │
  │  2. Server Hello                        │
  │  (选定的加密套件、随机数)                  │
  │  ◄───────────────────────────────────── │
  │                                         │
  │  3. Certificate                         │
  │  (服务器证书)                             │
  │  ◄───────────────────────────────────── │
  │                                         │
  │  4. 验证证书，生成预主密钥                 │
  │  用服务器公钥加密发送                      │
  │ ─────────────────────────────────────►  │
  │                                         │
  │  5. 服务器用私钥解密                      │
  │  双方生成会话密钥                         │
  │                                         │
  │  6. Finished                            │
  │  ◄─────────────────────────────────────►│
  │                                         │
  │  ========= 加密通信开始 =========         │
```

#### 加密过程详解

```
1. 对称加密 vs 非对称加密

对称加密：
┌─────────────────────────────────────────────────────────────────────────┐
│  • 加密和解密使用同一把密钥                                               │
│  • 速度快，适合大量数据                                                   │
│  • 问题：如何安全地交换密钥？                                             │
│  • 常见算法：AES、ChaCha20                                               │
└─────────────────────────────────────────────────────────────────────────┘

非对称加密：
┌─────────────────────────────────────────────────────────────────────────┐
│  • 公钥加密，私钥解密                                                    │
│  • 公钥可以公开，私钥保密                                                │
│  • 速度慢，适合少量数据                                                  │
│  • 常见算法：RSA、ECC                                                    │
└─────────────────────────────────────────────────────────────────────────┘

HTTPS 的混合加密方案：
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 握手阶段：使用非对称加密交换对称密钥                                   │
│  2. 通信阶段：使用对称加密传输数据                                        │
│  3. 兼顾安全性和性能                                                     │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 数字证书

```
证书作用：
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 身份认证：证明服务器身份                                              │
│  2. 公钥分发：安全地分发服务器公钥                                        │
│  3. 防止中间人攻击                                                       │
└─────────────────────────────────────────────────────────────────────────┘

证书链：
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                          │
│                    ┌─────────────┐                                      │
│                    │  根证书 CA   │ ← 操作系统/浏览器内置信任              │
│                    └──────┬──────┘                                      │
│                           │ 签发                                         │
│                    ┌──────▼──────┐                                      │
│                    │  中间证书 CA │ ← 帮助根证书分担签发工作               │
│                    └──────┬──────┘                                      │
│                           │ 签发                                         │
│                    ┌──────▼──────┐                                      │
│                    │  服务器证书  │ ← 域名验证，公钥                       │
│                    └─────────────┘                                      │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

证书验证过程：
1. 浏览器收到服务器证书
2. 查找签发该证书的 CA 证书
3. 用 CA 公钥验证签名
4. 逐级向上验证，直到根证书
5. 根证书在信任列表中，验证通过
```

### 3.4 证书类型

```
┌─────────────────┬───────────────────────────────────────────────────────┐
│  类型            │  说明                                                   │
├─────────────────┼───────────────────────────────────────────────────────┤
│  DV（域名验证）   │  只验证域名所有权，适合个人网站                          │
│                 │  签发速度快，价格便宜                                    │
├─────────────────┼───────────────────────────────────────────────────────┤
│  OV（组织验证）   │  验证组织真实性，显示公司名称                            │
│                 │  适合企业网站                                            │
├─────────────────┼───────────────────────────────────────────────────────┤
│  EV（扩展验证）   │  最严格验证，地址栏显示公司名称                          │
│                 │  适合金融、电商网站                                      │
├─────────────────┼───────────────────────────────────────────────────────┤
│  通配符证书      │  保护主域名及所有子域名                                  │
│                 │  *.example.com                                         │
├─────────────────┼───────────────────────────────────────────────────────┤
│  多域名证书      │  保护多个不同域名                                        │
│                 │  example.com, api.example.com, blog.example.org        │
└─────────────────┴───────────────────────────────────────────────────────┘
```

## 4. URL 结构解析

### 4.1 URL 组成

```
完整 URL 结构：
scheme://user:password@host:port/path?query#fragment

示例：
https://john:pass123@www.example.com:443/api/users?page=1&size=10#section1
└───┬───┘ └────┬────┘ └──────┬──────┘ └─┘ └────┬────┘ └───────┬───────┘ └───┬───┘
    │         │           │         │        │             │            │
   协议    用户信息       主机      端口     路径         查询参数       片段
```

### 4.2 各部分详解

```
┌───────────────────┬───────────────────────────────────────────────────────┐
│  组成部分          │  说明                                                   │
├───────────────────┼───────────────────────────────────────────────────────┤
│  scheme（协议）    │  使用的协议：http、https、ftp、ws、wss                   │
│                   │  必需部分                                               │
├───────────────────┼───────────────────────────────────────────────────────┤
│  user:password    │  认证信息（不推荐在 URL 中使用）                         │
│  （用户信息）      │  大多数浏览器已不支持                                    │
├───────────────────┼───────────────────────────────────────────────────────┤
│  host（主机）      │  服务器域名或 IP 地址                                    │
│                   │  www.example.com 或 192.168.1.1                         │
├───────────────────┼───────────────────────────────────────────────────────┤
│  port（端口）      │  服务器端口号                                            │
│                   │  默认：HTTP=80, HTTPS=443, FTP=21                        │
├───────────────────┼───────────────────────────────────────────────────────┤
│  path（路径）      │  资源在服务器上的路径                                    │
│                   │  /api/users                                             │
├───────────────────┼───────────────────────────────────────────────────────┤
│  query（查询）     │  键值对参数，? 开头，& 分隔                              │
│                   │  ?page=1&size=10                                        │
├───────────────────┼───────────────────────────────────────────────────────┤
│  fragment（片段）  │  页面内锚点，# 开头                                      │
│                   │  不发送到服务器                                          │
└───────────────────┴───────────────────────────────────────────────────────┘
```

### 4.3 URL 编码

```
为什么需要编码？
• URL 只能包含 ASCII 字符
• 特殊字符有特殊含义（?、&、=、#、/）
• 非安全字符需要编码

编码规则：
┌─────────────────┬───────────────────────────────────────────────────────┐
│  字符            │  编码后                                                 │
├─────────────────┼───────────────────────────────────────────────────────┤
│  空格            │  %20 或 +                                              │
│  中文            │  %E4%B8%AD%E6%96%87                                    │
│  &              │  %26                                                   │
│  =              │  %3D                                                   │
│  ?              │  %3F                                                   │
│  #              │  %23                                                   │
│  /              │  %2F                                                   │
└─────────────────┴───────────────────────────────────────────────────────┘

JavaScript 编码方法：
```javascript
// 编码
encodeURIComponent('你好世界');  // '%E4%BD%A0%E5%A5%BD%E4%B8%96%E7%95%8C'
encodeURI('https://example.com/你好');  // 'https://example.com/%E4%BD%A0%E5%A5%BD'

// 解码
decodeURIComponent('%E4%BD%A0%E5%A5%BD');  // '你好'
decodeURI('https://example.com/%E4%BD%A0%E5%A5%BD');  // 'https://example.com/你好'

// 区别：
// encodeURI：用于编码完整 URL，保留协议、域名中的特殊字符
// encodeURIComponent：用于编码 URL 参数，编码所有特殊字符
```

## 5. 常见问题与最佳实践

### 5.1 HTTP/1.1 优化策略

```javascript
// 1. 使用持久连接（默认开启）
// Connection: keep-alive

// 2. 合并资源减少请求数
// ❌ 多个小文件
<link rel="stylesheet" href="reset.css">
<link rel="stylesheet" href="base.css">
<link rel="stylesheet" href="theme.css">

// ✅ 合并为一个文件
<link rel="stylesheet" href="bundle.css">

// 3. 使用域名分片（已过时，HTTP/2 不需要）
// 浏览器对单个域名的并发连接数有限制（通常 6 个）
// 可以使用多个子域名绕过限制
// static1.example.com
// static2.example.com
// static3.example.com

// 4. 使用雪碧图减少图片请求
.icon {
  background-image: url('sprite.png');
  background-position: -16px -32px;
}
```

### 5.2 HTTPS 迁移注意事项

```
迁移步骤：
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 获取 SSL 证书                                                        │
│     • 选择合适的证书类型（DV/OV/EV）                                      │
│     • 可使用免费证书：Let's Encrypt                                      │
│                                                                          │
│  2. 配置服务器                                                            │
│     • 安装证书                                                           │
│     • 配置 HTTPS 监听（443 端口）                                         │
│                                                                          │
│  3. 设置 HTTP 到 HTTPS 重定向                                             │
│     • 301 永久重定向                                                     │
│     • 配置 HSTS 头部                                                     │
│                                                                          │
│  4. 更新内部资源链接                                                      │
│     • 确保所有资源使用 HTTPS                                              │
│     • 避免混合内容警告                                                    │
│                                                                          │
│  5. 更新外部链接                                                          │
│     • 更新 CDN、第三方服务链接                                            │
│     • 更新搜索引擎收录                                                    │
└─────────────────────────────────────────────────────────────────────────┘

Nginx 配置示例：
```nginx
# HTTP 重定向到 HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}

# HTTPS 配置
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
}
```

## 小结

### HTTP 版本特性速查

| 版本 | 年份 | 核心特性 | 主要改进 |
|------|------|----------|----------|
| HTTP/0.9 | 1991 | 仅 GET | 最简单的文本传输 |
| HTTP/1.0 | 1996 | POST/HEAD + 头部 | 支持多种数据类型 |
| HTTP/1.1 | 1997 | 持久连接 + 管道化 | 减少连接开销 |
| HTTP/2 | 2015 | 多路复用 + 头部压缩 | 解决队头阻塞 |
| HTTP/3 | 2022 | QUIC + 0-RTT | 更快连接建立 |

### HTTP vs HTTPS 速查

| 特性 | HTTP | HTTPS |
|------|------|-------|
| 端口 | 80 | 443 |
| 传输 | 明文 | 加密 |
| 安全 | 低 | 高 |
| 证书 | 不需要 | 需要 |
| 性能 | 快 | 略慢 |
