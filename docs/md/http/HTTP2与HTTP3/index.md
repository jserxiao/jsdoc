# HTTP/2 与 HTTP/3

> HTTP/2 和 HTTP/3 是 HTTP 协议的重大升级，引入了多路复用、头部压缩、服务器推送等新特性，大幅提升了 Web 性能。理解这些新特性有助于更好地优化 Web 应用。

## 学习要点

- 🔄 理解 HTTP/2 的核心特性
- 📦 掌握多路复用和头部压缩原理
- 🚀 了解 HTTP/3 与 QUIC 的创新
- 🎯 学会利用新特性优化性能

---

## 1. HTTP/2 概述

### 1.1 HTTP/1.1 的问题

```
HTTP/1.1 的性能瓶颈：
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 队头阻塞（Head-of-Line Blocking）                                    │
│     • 管道化请求必须按顺序响应                                           │
│     • 前面的请求延迟会阻塞后续请求                                        │
│                                                                          │
│  2. 连接数限制                                                           │
│     • 浏览器对单个域名有并发连接数限制（通常 6 个）                         │
│     • 需要域名分片来绕过限制                                             │
│                                                                          │
│  3. 头部冗余                                                             │
│     • 每次请求都发送完整头部                                             │
│     • 大量重复数据浪费带宽                                               │
│                                                                          │
│  4. 文本协议                                                             │
│     • 解析效率低                                                         │
│     • 容易出错                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.2 HTTP/2 核心特性

```
┌─────────────────────────────────────────────────────────────────────────┐
│  HTTP/2 核心特性                                                         │
├─────────────────────────┬───────────────────────────────────────────────┤
│  二进制分帧              │  将请求/响应分解为二进制帧，高效解析             │
├─────────────────────────┼───────────────────────────────────────────────┤
│  多路复用                │  单连接并行发送多个请求/响应                     │
├─────────────────────────┼───────────────────────────────────────────────┤
│  头部压缩（HPACK）       │  减少重复头部传输                               │
├─────────────────────────┼───────────────────────────────────────────────┤
│  服务器推送              │  服务器主动推送资源                             │
├─────────────────────────┼───────────────────────────────────────────────┤
│  请求优先级              │  设置请求权重和依赖关系                         │
├─────────────────────────┼───────────────────────────────────────────────┤
│  流量控制                │  控制数据传输速度                               │
└─────────────────────────┴───────────────────────────────────────────────┘
```

## 2. 二进制分帧

### 2.1 帧结构

```
HTTP/1.1 文本协议：
GET /index.html HTTP/1.1\r\n
Host: example.com\r\n
\r\n

HTTP/2 二进制帧：
┌───────────────────────────────────────────────────────────────┐
│                        Frame Header                            │
├───────────┬───────────┬───────────┬───────────┬───────────────┤
│  Length   │   Type    │   Flags   │  Stream Identifier        │
│  (24bit)  │  (8bit)   │  (8bit)   │  (31bit)                  │
├───────────┴───────────┴───────────┴───────────┴───────────────┤
│                        Frame Payload                           │
└───────────────────────────────────────────────────────────────┘
```

### 2.2 帧类型

```
┌─────────────────────┬───────────────────────────────────────────────────────┐
│  帧类型              │  说明                                                   │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  DATA (0x0)         │  传输请求或响应的实际数据                                 │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  HEADERS (0x1)      │  传输请求或响应头部                                      │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  PRIORITY (0x2)     │  指定流的优先级和依赖关系                                 │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  RST_STREAM (0x3)   │  终止流传输                                             │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  SETTINGS (0x4)     │  传输连接配置参数                                        │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  PUSH_PROMISE (0x5) │  服务器推送承诺                                          │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  PING (0x6)         │  心跳检测                                               │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  GOAWAY (0x7)       │  关闭连接                                               │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  WINDOW_UPDATE (0x8)│  流量控制窗口更新                                        │
├─────────────────────┼───────────────────────────────────────────────────────┤
│  CONTINUATION (0x9) │  继续传输头部块                                          │
└─────────────────────┴───────────────────────────────────────────────────────┘
```

### 2.3 流、消息、帧的关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│  层级关系                                                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  TCP 连接                                                                │
│    │                                                                     │
│    ├─── Stream 1（请求/响应）                                            │
│    │      │                                                              │
│    │      ├─── Message（请求）                                           │
│    │      │      ├─── Frame（HEADERS）                                   │
│    │      │      └─── Frame（DATA）                                      │
│    │      │                                                              │
│    │      └─── Message（响应）                                           │
│    │             ├─── Frame（HEADERS）                                   │
│    │             └─── Frame（DATA）                                      │
│    │                                                                     │
│    ├─── Stream 3（请求/响应）                                            │
│    │      └─── ...                                                       │
│    │                                                                     │
│    └─── Stream 5（请求/响应）                                            │
│           └─── ...                                                       │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

术语定义：
• Stream：双向字节流，承载一个或多个消息
• Message：完整的 HTTP 请求或响应
• Frame：最小的通信单位
```

## 3. 多路复用

### 3.1 对比 HTTP/1.1

```
HTTP/1.1（管道化，有队头阻塞问题）：
请求1 ──────────────────────────►
                               响应1 ◄─────────────────────────
     请求2 ──────────────────────────►
                                    响应2 ◄─────────────────────────

HTTP/2（多路复用，无队头阻塞）：
请求1 ──►
请求2 ──────►
请求3 ───►
      响应1 ◄──
      响应3 ◄───
      响应2 ◄─────

关键区别：
• HTTP/1.1：请求和响应必须串行
• HTTP/2：多个请求/响应可以交错传输
```

### 3.2 解决队头阻塞

```
HTTP/1.1 队头阻塞：
┌─────────────────────────────────────────────────────────────────────────┐
│  请求 1（慢响应）███████████████████████████████████████████████████     │
│  请求 2（快响应）等待...等待...等待...响应2 ████                          │
│  请求 3（快响应）等待...等待...等待...等待...响应3 ████                   │
│                                                                          │
│  问题：请求 1 延迟，阻塞了后续所有请求                                     │
└─────────────────────────────────────────────────────────────────────────┘

HTTP/2 多路复用：
┌─────────────────────────────────────────────────────────────────────────┐
│  请求 1 数据 ████████              ████████              ████████        │
│  请求 2 数据      ████████  ████████       ████████                     │
│  请求 3 数据           ████████       ████████       ████████           │
│                                                                          │
│  优势：各请求独立传输，互不影响                                           │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 单连接优势

```
HTTP/1.1 需要多连接：
┌─────────────────────────────────────────────────────────────────────────┐
│  浏览器对单个域名有并发连接数限制（通常 6 个）                              │
│  需要域名分片来绕过限制：                                                 │
│  • static1.example.com                                                  │
│  • static2.example.com                                                  │
│  • static3.example.com                                                  │
│                                                                          │
│  缺点：                                                                  │
│  • 增加连接开销                                                          │
│  • 增加服务器负担                                                        │
│  • DNS 查询增加                                                         │
│  • TCP/TLS 握手增加                                                     │
└─────────────────────────────────────────────────────────────────────────┘

HTTP/2 单连接：
┌─────────────────────────────────────────────────────────────────────────┐
│  单个 TCP 连接承载所有请求：                                              │
│  • 减少连接开销                                                          │
│  • 更好地利用 TCP                                                        │
│  • 更少的 TLS 握手                                                      │
│  • 不需要域名分片                                                        │
└─────────────────────────────────────────────────────────────────────────┘
```

## 4. 头部压缩（HPACK）

### 4.1 HTTP/1.1 的头部问题

```
HTTP/1.1（重复发送头部）：
请求1：
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html,...
Cookie: sessionId=abc123...

请求2：
GET /style.css HTTP/1.1
Host: example.com          ← 重复
User-Agent: Mozilla/5.0 ... ← 重复
Accept: text/css,...       ← 类似
Cookie: sessionId=abc123... ← 重复

问题：大量重复数据，浪费带宽
```

### 4.2 HPACK 压缩

```
HTTP/2（头部压缩）：
┌─────────────────────────────────────────────────────────────────────────┐
│  压缩机制：                                                              │
│  1. 静态字典：预定义常用头部                                              │
│  2. 动态字典：连接期间共享                                                │
│  3. Huffman 编码：压缩字符串                                              │
└─────────────────────────────────────────────────────────────────────────┘

静态表（常用头部预定义）：
┌───────┬─────────────────────────┬─────────────────────┐
│ 索引  │ 名称                    │ 值                   │
├───────┼─────────────────────────┼─────────────────────┤
│  1    │ :authority              │                      │
│  2    │ :method                 │ GET                  │
│  3    │ :method                 │ POST                 │
│  4    │ :path                   │ /                    │
│  5    │ :path                   │ /index.html          │
│ ...   │ ...                     │ ...                  │
│  44   │ accept-encoding         │ gzip, deflate        │
└───────┴─────────────────────────┴─────────────────────┘

动态表工作原理：
请求1：发送完整头部，存入动态表
请求2：使用索引引用之前的头部
请求3：只发送差异部分
```

### 4.3 压缩效果

```
示例对比：
┌─────────────────────────────────────────────────────────────────────────┐
│  HTTP/1.1 请求头部大小：约 800-1200 字节                                  │
│  HTTP/2 压缩后：约 50-200 字节                                           │
│  压缩率：80%-90%                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 5. 服务器推送

### 5.1 基本概念

```
传统方式：
客户端请求 index.html → 解析发现需要 style.css → 请求 style.css → 接收

服务器推送：
客户端请求 index.html
服务器响应 index.html + 主动推送 style.css

┌──────────┐                          ┌──────────┐
│  客户端   │                          │  服务器   │
└────┬─────┘                          └────┬─────┘
     │                                     │
     │  GET /index.html                    │
     │ ─────────────────────────────────►  │
     │                                     │
     │  响应 index.html                    │
     │  PUSH_PROMISE /style.css            │
     │  ◄────────────────────────────────  │
     │                                     │
     │  推送 style.css                     │
     │  ◄────────────────────────────────  │
     │                                     │
```

### 5.2 客户端处理

```javascript
// 检测服务器推送支持
if ('performance' in window && 'getEntriesByType' in performance) {
  const entries = performance.getEntriesByType('resource');
  // 可以查看资源加载方式
}

// 服务器推送的资源会被缓存
// 后续请求会直接使用缓存

// 拒绝推送（通过 settings 帧）
// 客户端可以禁用推送功能
```

### 5.3 替代方案

```http
# HTTP/2 服务器推送支持有限，现代方案：

# 1. Preload（预加载）
Link: </style.css>; rel=preload; as=style
Link: </script.js>; rel=preload; as=script

# 2. Early Hints（103）
HTTP/1.1 103 Early Hints
Link: </style.css>; rel=preload; as=style
Link: </script.js>; rel=preload; as=script

# 3. HTML 中的预加载
<link rel="preload" href="style.css" as="style">
<link rel="preload" href="script.js" as="script">
```

## 6. HTTP/3 与 QUIC

### 6.1 HTTP/2 的遗留问题

```
HTTP/2 解决了应用层的队头阻塞，但 TCP 层面仍有问题：
┌─────────────────────────────────────────────────────────────────────────┐
│  TCP 队头阻塞：                                                          │
│  • TCP 保证有序传输                                                      │
│  • 一个数据包丢失会阻塞后续所有数据包                                      │
│  • 即使这些数据包属于不同的流                                             │
│                                                                          │
│  示例：                                                                  │
│  数据包 1（流A） ✓ 到达                                                  │
│  数据包 2（流B） ✗ 丢失                                                  │
│  数据包 3（流C） ✓ 到达（但无法处理，等待包2）                             │
│  数据包 4（流A） ✓ 到达（但无法处理，等待包2）                             │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 QUIC 协议

```
QUIC（Quick UDP Internet Connections）：
┌─────────────────────────────────────────────────────────────────────────┐
│  基于 UDP 的可靠传输协议                                                  │
│                                                                          │
│  核心特性：                                                              │
│  1. 独立流传输                                                           │
│     • 每个流独立处理丢包                                                 │
│     • 一个流丢包不影响其他流                                             │
│                                                                          │
│  2. 0-RTT 连接建立                                                       │
│     • 复用之前的连接信息                                                 │
│     • 极快的连接恢复                                                    │
│                                                                          │
│  3. 连接迁移                                                             │
│     • 使用 Connection ID 而非四元组标识连接                               │
│     • 网络切换（WiFi→4G）不断开连接                                      │
│                                                                          │
│  4. 内置 TLS 1.3                                                        │
│     • 加密是协议的一部分                                                 │
│     • 减少握手开销                                                      │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.3 HTTP/3 vs HTTP/2

```
┌─────────────────┬───────────────────────────────────────────────────────┐
│  对比项          │  HTTP/2                │  HTTP/3                       │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  传输层          │  TCP                   │  QUIC (UDP)                   │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  队头阻塞        │  TCP 层面存在           │  完全解决                     │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  连接建立        │  TCP + TLS (2-3 RTT)   │  1-RTT / 0-RTT               │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  连接迁移        │  不支持                │  支持                        │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  加密            │  可选                  │  强制 (TLS 1.3)              │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  头部压缩        │  HPACK                 │  QPACK                       │
├─────────────────┼────────────────────────┼───────────────────────────────┤
│  浏览器支持      │  广泛                  │  逐渐增加                    │
└─────────────────┴────────────────────────┴───────────────────────────────┘
```

### 6.4 0-RTT 连接建立

```
首次连接（1-RTT）：
┌──────────┐                                            ┌──────────┐
│  客户端   │                                            │  服务器   │
└────┬─────┘                                            └────┬─────┘
     │  Client Hello + TLS 扩展                         │
     │ ─────────────────────────────────────────────►  │
     │                                                 │
     │  Server Hello + 证书 + 配置                      │
     │  ◄───────────────────────────────────────────── │
     │                                                 │
     │  ========= 开始传输数据 =========                │

后续连接（0-RTT）：
┌──────────┐                                            ┌──────────┐
│  客户端   │                                            │  服务器   │
└────┬─────┘                                            └────┬─────┘
     │  Client Hello + 0-RTT 数据                       │
     │ ─────────────────────────────────────────────►  │
     │                                                 │
     │  服务器处理请求                                   │
     │  ◄───────────────────────────────────────────── │
     │                                                 │
     │  ========= 立即开始传输 =========                │
```

## 7. 性能优化建议

### 7.1 HTTP/2 优化

```
不再需要的优化：
┌─────────────────────────────────────────────────────────────────────────┐
│  HTTP/1.1 优化              │  HTTP/2 状态                                │
├─────────────────────────────┼────────────────────────────────────────────┤
│  域名分片                    │  不再需要，反而有害                          │
│  合并文件                    │  不再必需，可以拆分                          │
│  内联资源                    │  不推荐，影响缓存                            │
│  图片精灵                    │  不再必需                                   │
└─────────────────────────────┴────────────────────────────────────────────┘

推荐的优化：
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 减少连接数                                                           │
│     • 使用单个域名                                                       │
│     • 让 HTTP/2 多路复用发挥作用                                         │
│                                                                          │
│  2. 合理拆分资源                                                         │
│     • 按功能模块拆分                                                     │
│     • 更好的缓存粒度                                                     │
│     • 只加载需要的资源                                                   │
│                                                                          │
│  3. 使用 preload/preconnect                                             │
│     • 提前建立连接                                                       │
│     • 预加载关键资源                                                     │
│                                                                          │
│  4. 优化资源优先级                                                       │
│     • CSS 优先加载                                                       │
│     • 关键 JS 尽早执行                                                   │
└─────────────────────────────────────────────────────────────────────────┘
```

### 7.2 服务端配置

```nginx
# Nginx HTTP/2 配置
server {
    listen 443 ssl http2;
    server_name example.com;
    
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    
    # HTTP/2 推送（如果需要）
    http2_push /style.css;
    http2_push /script.js;
    
    # 或使用 preload
    add_header Link "</style.css>; rel=preload; as=style";
}

# HTTP/3 配置（Nginx 1.25+）
server {
    listen 443 quic;
    listen 443 ssl;
    server_name example.com;
    
    add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

### 7.3 前端适配

```html
<!-- 使用 preload 预加载关键资源 -->
<link rel="preload" href="critical.css" as="style">
<link rel="preload" href="main.js" as="script">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

<!-- 使用 preconnect 提前建立连接 -->
<link rel="preconnect" href="https://cdn.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

<!-- 使用 modulepreload 预加载 ES 模块 -->
<link rel="modulepreload" href="module.js">
```

## 小结

### HTTP/2 关键特性

| 特性 | 说明 | 优势 |
|------|------|------|
| 二进制分帧 | 高效解析 | 性能提升 |
| 多路复用 | 单连接并行 | 解决队头阻塞 |
| 头部压缩 | HPACK | 减少带宽 |
| 服务器推送 | 主动推送 | 减少延迟 |

### HTTP/3 关键特性

| 特性 | 说明 | 优势 |
|------|------|------|
| QUIC 协议 | 基于 UDP | 可靠传输 |
| 0-RTT | 快速连接 | 降低延迟 |
| 连接迁移 | 网络切换 | 移动端优化 |
| 独立流 | 无队头阻塞 | 性能提升 |

### 版本对比

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 多路复用 | ✗ | ✓ | ✓ |
| 头部压缩 | ✗ | ✓ | ✓ |
| 传输层 | TCP | TCP | UDP |
| 队头阻塞 | 应用层+TCP | TCP | 无 |
